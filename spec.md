# Spec: ComfyUI Custom Node для Google Flow

**Название проекта:** `ComfyUI-GoogleFlow`  
**Версия спекa:** 1.0  
**Дата:** 24 июня 2026  
**Цель:** Интеграция Google Flow (labs.google/fx/tools/flow) в ComfyUI через browser automation для максимального использования подписки пользователя.

---

## 1. Обзор проекта

### 1.1 Что такое Google Flow
Google Flow — это AI Creative Studio от Google Labs, объединяющая генерацию и редактирование изображений (Nano Banana), видео (Veo 3.1) и работу с агентом (Gemini Omni). Доступ осуществляется через веб-интерфейс по подписке Google AI.

### 1.2 Проблема
У Flow **нет публичного API**. Все возможности доступны только через веб-интерфейс после авторизации Google-аккаунтом с активной подпиской.

### 1.3 Решение
Создать custom node для ComfyUI, который:
- Авторизуется в Google Flow через OAuth (Google аккаунт пользователя).
- Работает в **headless-режиме** после первого логина.
- Позволяет использовать генерацию и редактирование изображений.
- Поддерживает многошаговые сценарии через Agent mode.
- Максимально задействует квоты и возможности именно подписки пользователя.

## 2. Цели и ограничения

### Цели (MVP + Roadmap)
- **MVP**: Генерация и редактирование изображений через Nano Banana.
- Поддержка референсных изображений (image-to-image / subject reference).
- Полностью headless работа после первого логина.
- Возможность строить многошаговые итеративные сценарии.
- Сохранение сессии между запусками ComfyUI.

### Non-Goals (не в scope на первом этапе)
- Полноценная автоматизация видео (Veo 3.1) — позже.
- Автоматическое создание Custom Tools.
- Длинные автономные диалоги с Agent mode внутри одной ноды.
- Публичное использование / sharing workflow'ов.

## 3. Технический подход

### 3.1 Почему именно browser automation
- Нет официального API у Google Flow.
- Веб-интерфейс даёт доступ ко всем возможностям подписки (включая Agent mode и продвинутое редактирование).
- Playwright — самый стабильный инструмент для этой задачи.

### 3.2 Выбор инструмента
- **Playwright (Python)** — основной выбор.
  - Отличная поддержка persistent context.
  - Хорошая работа с Google OAuth.
  - Возможность headed + headless режимов.
  - Удобные селекторы и ожидания.

### 3.3 Стратегия работы с Agent mode
Полностью эмулировать conversational Agent mode внутри одной ноды — ненадёжно.  
Вместо этого:
- Делаем ноду `FlowAgentStep`, которая выполняет **один шаг** (отправляет изображение + инструкцию).
- Многошаговые сценарии пользователь строит сам в графе ComfyUI (цепочка нод + логика).

Это более стабильный и гибкий подход.

## 5. Аутентификация и сессия

### 5.1 Требования
- Первый запуск — `headed=True` (открывается браузер для логина).
- Все последующие запуски — `headed=False`.
- Сессия сохраняется в `user_data/` (cookies + localStorage).
- Автоматическое определение протухшей сессии.

### 5.2 Реализация
Класс `FlowAuthManager`:
- `ensure_session()` — проверяет/создаёт сессию.
- `login()` — запускает headed браузер для авторизации.
- `get_page()` — возвращает авторизованную страницу.
- Хранение контекста в `user_data/flow_context`.

**Рекомендация:** Использовать `playwright.sync_api` с `persistent_context`.

## 6. Функциональность (по приоритету)

### Приоритет 1: Изображения (Nano Banana)
- Генерация изображений по тексту.
- Редактирование изображений по текстовой инструкции.
- Использование reference image.
- Контроль силы редактирования (strength / guidance).

### Приоритет 2: Agent Mode (пошагово)
- Нода `FlowAgentStep`:
  - Вход: изображение + текстовая инструкция.
  - Выход: новое изображение + текстовый ответ агента (опционально).
- Возможность передавать контекст между шагами.

### Приоритет 3 (позже)
- Генерация видео (Veo 3.1).
- Более сложные многошаговые цепочки.

## 7. Дизайн нод

### 7.1 `FlowImageEdit` (самая важная нода MVP)
**Входы:**
- `image` (IMAGE)
- `instruction` (STRING) — что нужно сделать
- `strength` (FLOAT, 0.0–1.0)
- `model` (DROPDOWN) — если будет выбор

**Выходы:**
- `image` (IMAGE)
- `info` (STRING) — текстовый ответ от Flow (опционально)

### 7.2 `FlowAgentStep`
**Входы:**
- `image` (IMAGE)
- `instruction` (STRING)
- `context` (STRING, optional) — предыдущий ответ агента

**Выходы:**
- `image` (IMAGE)
- `response` (STRING)

---

## 8. Реализация многошаговых сценариев

Пользователь строит сценарий в ComfyUI следующим образом:

1. `Load Image` → `FlowImageEdit` (шаг 1)
2. `FlowAgentStep` (шаг 2)
3. `FlowImageEdit` (шаг 3)
4. ... и так далее

Это даёт полный контроль и прозрачность.

## 9. Технические вызовы и решения

| Вызов | Решение | Приоритет |
|-------|---------|---------|
| Изменение вёрстки Flow | Использовать устойчивые селекторы + fallback | Высокий |
| Долгое время генерации | `wait_for` + увеличенные таймауты + polling | Высокий |
| Headless + Google login | Persistent context + user_data | Высокий |
| Agent mode (диалоговый) | Делать пошагово через несколько нод | Средний |
| Обработка ошибок | Красивые сообщения + попытка перелогина | Средний |
| Производительность | Запускать браузер один раз на сессию ComfyUI | Средний |

## 11. Ограничения и риски

- **Неофициальная интеграция** — Google может изменить интерфейс в любой момент.
- Квоты и лимиты полностью зависят от подписки пользователя в Google Flow.
- Видео-генерация и тяжёлые Agent-сценарии могут работать медленно.
- Полная автоматизация Agent mode (длинные разговоры) — нестабильна.
- Для личного использования риски приемлемы.
